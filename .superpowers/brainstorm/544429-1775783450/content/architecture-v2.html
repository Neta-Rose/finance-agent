Architecture v2 — Proxy as Universal Enforcement Point

The proxy now intercepts ALL agent activity — dashboard AND Telegram AND cron.

Why the proxy is the perfect choke point

🖥️
Dashboard
User clicks "Trigger Job"
→ POST /api/jobs/trigger
📱
Telegram
User sends message
→ OpenClaw starts session
⏰
Scheduled Cron
8am daily brief
→ OpenClaw starts session
All three paths converge here ↓
POST /llm/v1/chat/completions
Every LLM call from every agent passes through here.
This is where we enforce blocks — before ANY token reaches OpenRouter.
✓ User not blocked
Forward to OpenRouter → agent gets response → job completes
✗ User is blocked/suspended
Return 403 {"error":"Account suspended. Contact admin."}
Agent fails gracefully — job marked failed by watchdog

Enforcement layers (defense in depth)

LayerWhat it blocksCovers
LLM Proxy All LLM calls — the agent can't do anything without a model response Dashboard + Telegram + Cron
jobs.ts route Dashboard "trigger job" buttons — blocked before a trigger file is even written Dashboard only
userControlMiddleware GET /api/me/control — tells frontend what state to render (suspension page, banner) Dashboard UX
authMiddleware tokenVersion check — force-logout invalidates all active sessions immediately All API calls

What "blocked" means at the proxy level

none
Proxy forwards normally to OpenRouter
readonly
Proxy blocks → 403. Dashboard shows info banner. No new jobs from any source.
blocked
Proxy blocks → 403. Dashboard shows warning. Telegram triggers die silently.
suspended
Proxy blocks → 403. Login works but lands on suspension page. No dashboard.
