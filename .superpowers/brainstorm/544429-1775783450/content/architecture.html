Architecture — Admin Controls

How all the pieces connect. Review and tell me if anything looks wrong.

Storage: Two control files

Per-user
users/{userId}/control.json
  • restriction: null | "readonly" | "blocked" | "suspended"
  • reason: admin message shown to user
  • restrictedAt: ISO timestamp
  • restrictedUntil: ISO or null (indefinite)
  • banner: { text, type, dismissible, expiresAt }
  • tokenVersion: incremented on force-logout
  • activeJobKills: list of job IDs to force-fail
System-wide
data/system-control.json
  • locked: boolean
  • lockReason: shown to all users
  • lockedAt / lockedUntil
  • broadcast: { text, type, dismissible, expiresAt }

Server-side enforcement (security-first)

Request → authMiddleware validates JWT + checks tokenVersion
→ systemLockMiddleware blocks all if system locked (except admin routes)
→ userControlMiddleware reads control.json, attaches restriction to res.locals
→ GET /api/me/control frontend polls this to know current state — ALWAYS responds
→ POST /api/jobs/trigger blocked/readonly/suspended → 403 with code

Security guarantees

Enforced server-side
  • All blocks checked per-request, not just login
  • tokenVersion in JWT payload — can't be forged
  • System lock bypasses user workspace entirely
  • Admin endpoints still use X-Admin-Key only (no JWT)
  • control.json only writable by backend, never client
  • Auto-expiry evaluated server-side on read
What this doesn't do
  • Doesn't block the user's Telegram access
    (separate OpenClaw channel — out of scope)
  • Doesn't revoke existing OpenClaw LLM proxy keys
    (proxy key revocation is handled via unregisterKey)
  • Rate limit override zeroes the window count, not forever
    (resets when their rate limit window rolls over)
